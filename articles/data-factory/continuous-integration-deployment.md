<p>[!INCLUDE<a href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/includes/appliesto-adf-xxx-md.md">appliesto-adf-xxx-md</a>]</p>
<h2><a id="user-content-overview" class="anchor" aria-hidden="true" href="#overview"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Overview</h2>
<p>Continuous integrationÂ is the practice of testing each change made to your codebase automatically and as early as possible.Â Continuous deliveryÂ follows the testing that happens duringÂ continuous integration and pushes changes to a staging or production system.</p>
<p>In Azure Data Factory, continuous integration and delivery (CI/CD) means moving Data Factory pipelines from one environment (development, test, production) to another. Azure Data Factory utilizes <a href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/azure-resource-manager/templates/overview.md">Azure Resource Manager templates</a> to store the configuration of your various ADF entities (pipelines, datasets, data flows, and so on). There are two suggested methods to promote a data factory to another environment:</p>
<ul>
<li>Automated deployment using Data Factory's integration with <a href="/MicrosoftDocs/azure-docs-pr/blob/master/azure/devops/pipelines/get-started/what-is-azure-pipelines">Azure Pipelines</a></li>
<li>Manually upload a Resource Manager template using Data Factory UX integration with Azure Resource Manager.</li>
</ul>
<p>[!INCLUDE <a href="/MicrosoftDocs/azure-docs-pr/blob/master/includes/updated-for-az.md">updated-for-az</a>]</p>
<h2><a id="user-content-cicd-lifecycle" class="anchor" aria-hidden="true" href="#cicd-lifecycle"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>CI/CD lifecycle</h2>
<p>Below is a sample overview of the CI/CD lifecycle in an Azure data factory that's configured with Azure Repos Git. For more information on how to configure a Git repository, see <a href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/source-control.md">Source control in Azure Data Factory</a>.</p>
<ol>
<li>
<p>A development data factory is created and configured with Azure Repos Git. All developers should have permission to author Data Factory resources like pipelines and datasets.</p>
</li>
<li>
<p>A developer <a href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/source-control.md#creating-feature-branches">creates a feature branch</a> to make a change. They debug their pipeline runs with their most recent changes. For more information on how to debug a pipeline run, see <a href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/iterative-development-debugging.md">Iterative development and debugging with Azure Data Factory</a>.</p>
</li>
<li>
<p>After a developer is satisfied with their changes, they create a pull request from their feature branch to the main or collaboration branch to get their changes reviewed by peers.</p>
</li>
<li>
<p>After a pull request is approved and changes are merged in the main branch, the changes get published to the development factory.</p>
</li>
<li>
<p>When the team is ready to deploy the changes to a test or UAT (User Acceptance Testing) factory, the team goes to their Azure Pipelines release and deploys the desired version of the development factory to UAT. This deployment takes place as part of an Azure Pipelines task and uses Resource Manager template parameters to apply the appropriate configuration.</p>
</li>
<li>
<p>After the changes have been verified in the test factory, deploy to the production factory by using the next task of the pipelines release.</p>
</li>
</ol>
<blockquote>
<p>[!NOTE]
Only the development factory is associated with a git repository. The test and production factories shouldn't have a git repository associated with them and should only be updated via an Azure DevOps pipeline or via a Resource Management template.</p>
</blockquote>

<ol>
<li>
<p>In <a href="https://dev.azure.com/" rel="nofollow">Azure DevOps</a>, open the project that's configured with your data factory.</p>
</li>
<li>
<p>On the left side of the page, select <strong>Pipelines</strong>, and then select <strong>Releases</strong>.</p>
<p><a target="_blank" rel="noopener noreferrer" href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/media/continuous-integration-deployment/continuous-integration-image6.png"><img src="/MicrosoftDocs/azure-docs-pr/raw/master/articles/data-factory/media/continuous-integration-deployment/continuous-integration-image6.png" alt="Select Pipelines, Releases" style="max-width:100%;"></a></p>
</li>
<li>
<p>Select <strong>New pipeline</strong>, or, if you have existing pipelines, select <strong>New</strong> and then <strong>New release pipeline</strong>.</p>
</li>
<li>
<p>Select the <strong>Empty job</strong> template.</p>
<p><a target="_blank" rel="noopener noreferrer" href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/media/continuous-integration-deployment/continuous-integration-image13.png"><img src="/MicrosoftDocs/azure-docs-pr/raw/master/articles/data-factory/media/continuous-integration-deployment/continuous-integration-image13.png" alt="Select Empty job" style="max-width:100%;"></a></p>
</li>
<li>
<p>In the <strong>Stage name</strong> box, enter the name of your environment.</p>
</li>
<li>
<p>Select <strong>Add artifact</strong>, and then select the git repository configured with your development data factory. Select the <a href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/source-control.md#configure-publishing-settings">publish branch</a> of the repository for the <strong>Default branch</strong>. By default, this publish branch is <code>adf_publish</code>. For the <strong>Default version</strong>, select <strong>Latest from default branch</strong>.</p>
<p><a target="_blank" rel="noopener noreferrer" href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/media/continuous-integration-deployment/continuous-integration-image7.png"><img src="/MicrosoftDocs/azure-docs-pr/raw/master/articles/data-factory/media/continuous-integration-deployment/continuous-integration-image7.png" alt="Add an artifact" style="max-width:100%;"></a></p>
</li>
<li>
<p>Add an Azure Resource Manager Deployment task:</p>
<p>a.  In the stage view, select <strong>View stage tasks</strong>.</p>
<p><a target="_blank" rel="noopener noreferrer" href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/media/continuous-integration-deployment/continuous-integration-image14.png"><img src="/MicrosoftDocs/azure-docs-pr/raw/master/articles/data-factory/media/continuous-integration-deployment/continuous-integration-image14.png" alt="Stage view" style="max-width:100%;"></a></p>
<p>b.  Create a new task. Search for <strong>ARM Template Deployment</strong>, and then select <strong>Add</strong>.</p>
<p>c.  In the Deployment task, select the subscription, resource group, and location for the target data factory. Provide credentials if necessary.</p>
<p>d.  In the <strong>Action</strong> list, select <strong>Create or update resource group</strong>.</p>
<p>e.  Select the ellipsis button (<strong>â¦</strong>) next to the <strong>Template</strong> box. Browse for the Azure Resource Manager template that is generated in your publish branch of the configured git repository. Look for the file <code>ARMTemplateForFactory.json</code> in the  folder of the adf_publish branch.</p>
<p>f.  Select <strong>â¦</strong> next to the <strong>Template parameters</strong> box to choose the parameters file. Look for the file <code>ARMTemplateParametersForFactory.json</code> in the  folder of the adf_publish branch.</p>
<p>g.  Select <strong>â¦</strong> next to the <strong>Override template parameters</strong> box, and enter the desired parameter values for the target data factory. For credentials that come from Azure Key Vault, enter the secret's name between double quotation marks. For more information, please refer to <a href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/azure-resource-manager/templates/deployment-modes.md">Azure Resource Manager Deployment Modes</a></p>

<p><a target="_blank" rel="noopener noreferrer" href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/media/continuous-integration-deployment/continuous-integration-image9.png"><img src="/MicrosoftDocs/azure-docs-pr/raw/master/articles/data-factory/media/continuous-integration-deployment/continuous-integration-image9.png" alt="Data Factory Prod Deployment" style="max-width:100%;"></a></p>

<h3><a id="user-content-get-secrets-from-azure-key-vault" class="anchor" aria-hidden="true" href="#get-secrets-from-azure-key-vault"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Get secrets from Azure Key Vault</h3>
<p>If you have secrets to pass in an Azure Resource Manager template, we recommend that you use Azure Key Vault with the Azure Pipelines release.</p>
<p>There are two ways to handle secrets:</p>
<ol>
<li>
<p>Add the secrets to parameters file. For more info, see <a href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/azure-resource-manager/templates/key-vault-parameter.md">Use Azure Key Vault to pass secure parameter value during deployment</a>.</p>
<p>Create a copy of the parameters file that's uploaded to the publish branch. Set the values of the parameters that you want to get from Key Vault by using this format:</p>
<div class="highlight highlight-source-json"><pre>{
    <span class="pl-s"><span class="pl-pds">"</span>parameters<span class="pl-pds">"</span></span>: {
        <span class="pl-s"><span class="pl-pds">"</span>azureSqlReportingDbPassword<span class="pl-pds">"</span></span>: {
            <span class="pl-s"><span class="pl-pds">"</span>reference<span class="pl-pds">"</span></span>: {
                <span class="pl-s"><span class="pl-pds">"</span>keyVault<span class="pl-pds">"</span></span>: {
                    <span class="pl-s"><span class="pl-pds">"</span>id<span class="pl-pds">"</span></span>: <span class="pl-s"><span class="pl-pds">"</span>/subscriptions/&lt;subId&gt;/resourceGroups/&lt;resourcegroupId&gt; /providers/Microsoft.KeyVault/vaults/&lt;vault-name&gt; <span class="pl-pds">"</span></span>
                },
                <span class="pl-s"><span class="pl-pds">"</span>secretName<span class="pl-pds">"</span></span>: <span class="pl-s"><span class="pl-pds">"</span> &lt; secret - name &gt; <span class="pl-pds">"</span></span>
            }
        }
    }
}</pre></div>
<p>When you use this method, the secret is pulled from the key vault automatically.</p>
<p>The parameters file needs to be in the publish branch as well.</p>
</li>
<li>
<p>Add an <a href="/MicrosoftDocs/azure-docs-pr/blob/master/azure/devops/pipelines/tasks/deploy/azure-key-vault">Azure Key Vault task</a> before the Azure Resource Manager Deployment task described in the previous section:</p>
<ol>
<li>
<p>On the <strong>Tasks</strong> tab, create a new task. Search for <strong>Azure Key Vault</strong> and add it.</p>
</li>
<li>
<p>In the Key Vault task, select the subscription in which you created the key vault. Provide credentials if necessary, and then select the key vault.</p>
</li>
</ol>
<p><a target="_blank" rel="noopener noreferrer" href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/media/continuous-integration-deployment/continuous-integration-image8.png"><img src="/MicrosoftDocs/azure-docs-pr/raw/master/articles/data-factory/media/continuous-integration-deployment/continuous-integration-image8.png" alt="Add a Key Vault task" style="max-width:100%;"></a></p>
</li>
</ol>
<h4><a id="user-content-grant-permissions-to-the-azure-pipelines-agent" class="anchor" aria-hidden="true" href="#grant-permissions-to-the-azure-pipelines-agent"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Grant permissions to the Azure Pipelines agent</h4>
<p>The Azure Key Vault task might fail with an Access Denied error if the correct permissions aren't set. Download the logs for the release, and locate the .ps1 file that contains the command to give permissions to the Azure Pipelines agent. You can run the command directly. Or you can copy the principal ID from the file and add the access policy manually in the Azure portal. <code>Get</code> and <code>List</code> are the minimum permissions required.</p>
<h3><a id="user-content-updating-active-triggers" class="anchor" aria-hidden="true" href="#updating-active-triggers"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Updating active triggers</h3>
<p>Deployment can fail if you try to update active triggers. To update active triggers, you need to manually stop them and then restart them after the deployment. You can do this by using an Azure PowerShell task:</p>
<ol>
<li>
<p>On the <strong>Tasks</strong> tab of the release, add an <strong>Azure PowerShell</strong> task. Choose task version 4.*.</p>
</li>
<li>
<p>Select the subscription your factory is in.</p>
</li>
<li>
<p>Select <strong>Script File Path</strong> as the script type. This requires you to save your PowerShell script in your repository. The following PowerShell script can be used to stop triggers:</p>
<div class="highlight highlight-source-powershell"><pre><span class="pl-smi">$triggersADF</span> <span class="pl-k">=</span> <span class="pl-c1">Get-AzDataFactoryV2Trigger</span> <span class="pl-k">-</span>DataFactoryName <span class="pl-smi">$DataFactoryName</span> <span class="pl-k">-</span>ResourceGroupName <span class="pl-smi">$ResourceGroupName</span>

<span class="pl-smi">$triggersADF</span> <span class="pl-k">|</span> <span class="pl-c1">ForEach-Object</span> { <span class="pl-c1">Stop-AzDataFactoryV2Trigger</span> <span class="pl-k">-</span>ResourceGroupName <span class="pl-smi">$ResourceGroupName</span> <span class="pl-k">-</span>DataFactoryName <span class="pl-smi">$DataFactoryName</span> <span class="pl-k">-</span>Name <span class="pl-c1">$_<span class="pl-smi">.name</span></span> <span class="pl-k">-</span>Force }</pre></div>
</li>
</ol>
<p>You can complete similar steps (with the <code>Start-AzDataFactoryV2Trigger</code> function) to restart the triggers after deployment.</p>
<p>The data factory team has provided a <a href="#script">sample pre- and post-deployment script</a> located at the bottom of this article.</p>

<h2><a id="user-content-use-custom-parameters-with-the-resource-manager-template" class="anchor" aria-hidden="true" href="#use-custom-parameters-with-the-resource-manager-template"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Use custom parameters with the Resource Manager template</h2>
<p>If your development factory has an associated git repository, you can override the default Resource Manager template parameters of the Resource Manager template generated by publishing or exporting the template. You might want to override the default parameterization template in these scenarios:</p>
<ul>
<li>
<p>You use automated CI/CD and you want to change some properties during Resource Manager deployment, but the properties aren't parameterized by default.</p>
</li>
<li>
<p>Your factory is so large that the default Resource Manager template is invalid because it has more than the maximum allowed parameters (256).</p>
<p>To handle custom parameter 256 limit, there are 3 options:</p>
<ul>
<li>Use the custom parameter file and remove properties that don't need parameterization, i.e., properties that can keep a default value and hence decrease the parameter count.</li>
<li>Refactor logic in the dataflow to reduce parameters, for example, pipeline parameters all have the same value, you can just use global parameters instead.</li>
<li>Split one data factory  into multiple data flows.</li>
</ul>
</li>
</ul>
<p>To override the default parameterization template, go to the management hub and select <strong>Parameterization template</strong> in the source control section. Select <strong>Edit template</strong> to open the parameterization template code editor.</p>
<p><a target="_blank" rel="noopener noreferrer" href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/media/author-management-hub/management-hub-custom-parameters.png"><img src="/MicrosoftDocs/azure-docs-pr/raw/master/articles/data-factory/media/author-management-hub/management-hub-custom-parameters.png" alt="Manage custom parameters" style="max-width:100%;"></a></p>
<p>Creating a custom parameterization template creates a file named <strong>arm-template-parameters-definition.json</strong> in the root folder of your git branch. You must use that exact file name.</p>
<p><a target="_blank" rel="noopener noreferrer" href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/media/continuous-integration-deployment/custom-parameters.png"><img src="/MicrosoftDocs/azure-docs-pr/raw/master/articles/data-factory/media/continuous-integration-deployment/custom-parameters.png" alt="Custom parameters file" style="max-width:100%;"></a></p>
<p>When publishing from the collaboration branch, Data Factory will read this file and use its configuration to generate which properties get parameterized. If no file is found, the default template is used.</p>
<p>When exporting a Resource Manager template, Data Factory reads this file from whichever branch you're currently working on, not the collaboration branch. You can create or edit the file from a private branch, where you can test your changes by selecting <strong>Export ARM Template</strong> in the UI. You can then merge the file into the collaboration branch.</p>
<blockquote>
<p>[!NOTE]
A custom parameterization template doesn't change the ARM template parameter limit of 256. It lets you choose and decrease the number of parameterized properties.</p>
</blockquote>
<h3><a id="user-content-custom-parameter-syntax" class="anchor" aria-hidden="true" href="#custom-parameter-syntax"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Custom parameter syntax</h3>
<p>The following are some guidelines to follow when you create the custom parameters file, <strong>arm-template-parameters-definition.json</strong>. The file consists of a section for each entity type: trigger, pipeline, linked service, dataset, integration runtime, and data flow.</p>
<ul>
<li>Enter the property path under the relevant entity type.</li>
<li>Setting a property name toÂ <code>*</code> indicates that you want to parameterize all properties under it (only down to the first level, not recursively). You can also provide exceptions to this configuration.</li>
<li>Setting the value of a property as a string indicates that you want to parameterize the property. Use the formatÂ <code>&lt;action&gt;:&lt;name&gt;:&lt;stype&gt;</code>.
<ul>
<li><code>&lt;action&gt;</code>Â can be one of these characters:
<ul>
<li><code>=</code>Â means keep the current value as the default value for the parameter.</li>
<li><code>-</code>Â means don't keep the default value for the parameter.</li>
<li><code>|</code>Â is a special case for secrets from Azure Key Vault for connection strings or keys.</li>
</ul>
</li>
<li><code>&lt;name&gt;</code>Â is the name of the parameter. IfÂ it's blank, it takes the name of the property. If the value starts with a <code>-</code> character, the name is shortened. For example, <code>AzureStorage1_properties_typeProperties_connectionString</code> would be shortened to <code>AzureStorage1_connectionString</code>.</li>
<li><code>&lt;stype&gt;</code>Â is the type of parameter. IfÂ <code>&lt;stype&gt;</code>Â is blank, the default type is <code>string</code>. Supported values: <code>string</code>, <code>securestring</code>, <code>int</code>, <code>bool</code>, <code>object</code>, <code>secureobject</code> and <code>array</code>.</li>
</ul>
</li>
<li>Specifying an array in the definition file indicates that the matching property in the template is an array. Data Factory iterates through all the objects in the array by using the definition that's specified in the integration runtime object of the array. The second object, a string, becomes the name of the property, which is used as the name for the parameter for each iteration.</li>
<li>A definition can't be specific to a resource instance. Any definition applies to all resources of that type.</li>
<li>By default, all secure strings, like Key Vault secrets, and secure strings, like connection strings, keys, and tokens, are parameterized.</li>
</ul>

<h4><a id="user-content-pipelines" class="anchor" aria-hidden="true" href="#pipelines"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Pipelines</h4>
<ul>
<li>Any property in the path <code>activities/typeProperties/waitTimeInSeconds</code> is parameterized. Any activity in a pipeline that has a code-level property named <code>waitTimeInSeconds</code> (for example, the <code>Wait</code> activity) is parameterized as a number, with a default name. <li>Similarly, a property called <code>headers</code> (for example, in a <code>Web</code> activity) is parameterized with type <code>object</code> (JObject). It has a default value, which is the same value as that of the source factory.</li>

<ul>
<li>All properties under the path <code>typeProperties</code> are parameterized with their respective default values. <h4><a id="user-content-triggers" class="anchor" aria-hidden="true" href="#triggers"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Triggers</h4>
<ul>
<li>Under <code>typeProperties</code>, two properties are parameterized. The first one is <code>maxConcurrency</code>, which is specified to have a default value and is of type<code>string</code>. It has the default parameter name <code>&lt;entityName&gt;_properties_typeProperties_maxConcurrency</code>.</li>
<li>The <code>recurrence</code> property also is parameterized. Under it, all properties at that level are specified to be parameterized as strings, with default values and parameter names. The parameter name is suffixed with <code>&lt;entityName&gt;_properties_typeProperties_recurrence_triggerSuffix</code>. Similarly, the <code>freq</code> property is a string and is parameterized as a string. However, the <code>freq</code> property is parameterized without a default value. The name is shortened and suffixed. For example, <code>&lt;entityName&gt;_freq</code>.</li>
</ul>
<h4><a id="user-content-linkedservices" class="anchor" aria-hidden="true" href="#linkedservices"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>LinkedServices</h4>
<ul>
<li>Linked services are unique. Because linked services and datasets have a wide range of types, you can provide type-specific customization. <li>The <code>connectionString</code> property will be parameterized as a <code>securestring</code> value. It won't have a default value. It will have a shortened parameter name that's suffixed with <code>connectionString</code>.</li>
<li>The property <code>secretAccessKey</code> happens to be an <code>AzureKeyVaultSecret</code> (for example, in an Amazon S3 linked service). </ul>

<h2><a id="user-content-linked-resource-manager-templates" class="anchor" aria-hidden="true" href="#linked-resource-manager-templates"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Linked Resource Manager templates</h2>
<p>If you've set up CI/CD for your data factories, you might exceed the Azure Resource Manager template limits as your factory grows bigger. For example, one limit is the maximum number of resources in a Resource Manager template. To accommodate large factories while generating the full Resource Manager template for a factory, Data Factory now generates linked Resource Manager templates. With this feature, the entire factory payload is broken down into several files so that you aren't constrained by the limits.</p>
<p>If you've configured Git, the linked templates are generated and saved alongside the full Resource Manager templates in the adf_publish branch in a new folder called linkedTemplates:</p>
<p><a target="_blank" rel="noopener noreferrer" href="/MicrosoftDocs/azure-docs-pr/blob/master/articles/data-factory/media/continuous-integration-deployment/linked-resource-manager-templates.png"><img src="/MicrosoftDocs/azure-docs-pr/raw/master/articles/data-factory/media/continuous-integration-deployment/linked-resource-manager-templates.png" alt="Linked Resource Manager templates folder" style="max-width:100%;"></a></p>
<p>The linked Resource Manager templates usually consist of a master template and a set of child templates that are linked to the master. The parent template is called ArmTemplate_master.json, and child templates are named with the pattern ArmTemplate_0.json, ArmTemplate_1.json, and so on.</p>
<p>To use linked templates instead of the full Resource Manager template, update your CI/CD task to point to ArmTemplate_master.json instead of ArmTemplateForFactory.json (the full Resource Manager template). Resource Manager also requires that you upload the linked templates into a storage account so Azure can access them during deployment. For more info, see <a href="/MicrosoftDocs/azure-docs-pr/blob/master/archive/blogs/najib/deploying-linked-arm-templates-with-vsts">Deploying linked Resource Manager templates with VSTS</a>.</p>
<p>Remember to add the Data Factory scripts in your CI/CD pipeline before and after the deployment task.</p>
<p>If you don't have Git configured, you can access the linked templates via <strong>Export ARM Template</strong> in the <strong>ARM Template</strong> list.</p>

<p>If you deploy a factory to production and realize there's a bug that needs to be fixed right away, but you can't deploy the current collaboration branch, you might need to deploy a hotfix. This approach is as known as quick-fix engineering or QFE.</p>
<ol>
<li>
<p>In Azure DevOps, go to the release that was deployed to production. Find the last commit that was deployed.</p>
</li>
<li>
<p>From the commit message, get the commit ID of the collaboration branch.</p>
</li>
<li>
<p>Create a new hotfix branch from that commit.</p>
</li>
<li>
<p>Go to the Azure Data Factory UX and switch to the hotfix branch.</p>
</li>
<li>
<p>By using the Azure Data Factory UX, fix the bug. Test your changes.</p>
</li>
<li>
<p>After the fix is verified, select <strong>Export ARM Template</strong> to get the hotfix Resource Manager template.</p>
</li>
<li>
<p>Manually check this build into the adf_publish branch.</p>
</li>
<li>
<p>If you've configured your release pipeline to automatically trigger based on adf_publish check-ins, a new release will start automatically. Otherwise, manually queue a release.</p>
</li>
<li>
<p>Deploy the hotfix release to the test and production factories. This release contains the previous production payload plus the fix that you made in step 5.</p>
</li>
<li>
<p>Add the changes from the hotfix to the development branch so that later releases won't include the same bug.</p>
</li>
</ol>

<h2><a id="user-content-best-practices-for-cicd" class="anchor" aria-hidden="true" href="#best-practices-for-cicd"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Best practices for CI/CD</h2>
<p>If you're using Git integration with your data factory and have a CI/CD pipeline that moves your changes from development into test and then to production, we recommend these best practices:</p>
<ul>
<li>
<p><strong>Git integration</strong>. Configure only your development data factory with Git integration. Changes to test and production are deployed via CI/CD and don't need Git integration.</p>
</li>
<li>
<p><strong>Pre- and post-deployment script</strong>. Before the Resource Manager deployment step in CI/CD, you need to complete certain tasks, like stopping and restarting triggers and performing cleanup. We recommend that you use PowerShell scripts before and after the deployment task. For more information, see <a href="#updating-active-triggers">Update active triggers</a>. The data factory team has <a href="#script">provided a script</a> to use located at the bottom of this page.</p>
</li>
<li>
<p><strong>Integration runtimes and sharing</strong>. Integration runtimes don't change often and are similar across all stages in your CI/CD. So Data Factory expects you to have the same name and type of integration runtime across all stages of CI/CD. If you want to share integration runtimes across all stages, consider using a ternary factory just to contain the shared integration runtimes. You can use this shared factory in all of your environments as a linked integration runtime type.</p>
</li>
<li>
<p><strong>Managed private endpoint deployment</strong>. If a private endpoint already exists in a factory and you try to deploy an ARM template that contains a private endpoint with the same name but with modified properties, the deployment will fail. In other words, you can successfully deploy a private endpoint as long as it has the same properties as the one that already exists in the factory. If any property is different between environments, you can override it by parameterizing that property and providing the respective value during deployment.</p>
</li>
<li>
<p><strong>Key Vault</strong>. When you use linked services whose connection information is stored in Azure Key Vault, it is recommended to keep separate key vaults for different environments. You can also configure separate permission levels for each key vault. For example, you might not want your team members to have permissions to production secrets. If you follow this approach, we recommend that you to keep the same secret names across all stages. If you keep the same secret names, you don't need to parameterize each connection string across CI/CD environments because the only thing that changes is the key vault name, which is a separate parameter.</p>
</li>
<li>
<p><strong>Resource naming</strong> Due to ARM template constraints, issues in deployment may arise if your resources contain spaces in the name. The Azure Data Factory team recommends using '_' or '-' characters instead of spaces for resources. For example, 'Pipeline_1' would be a preferable name over 'Pipeline 1'.</p>
</li>
</ul>
<h2><a id="user-content-unsupported-features" class="anchor" aria-hidden="true" href="#unsupported-features"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Unsupported features</h2>
<ul>
<li>
<p>By design, Data Factory doesn't allow cherry-picking of commits or selective publishing of resources. Publishes will include all changes made in the data factory.</p>
<ul>
<li>Data factory entities depend on each other. For example, triggers depend on pipelines, and pipelines depend on datasets and other pipelines. Selective publishing of a subset of resources could lead to unexpected behaviors and errors.</li>
<li>On rare occasions when you need selective publishing, consider using a hotfix. For more information, see <a href="#hotfix-production-environment">Hotfix production environment</a>.</li>
</ul>
</li>
<li>
<p>The Azure Data Factory team doesnât recommend assigning Azure RBAC controls to individual entities (pipelines, datasets, etc.) in a data factory. For example, if a developer has access to a pipeline or a dataset, they should be able to access all pipelines or datasets in the data factory. If you feel that you need to implement many Azure roles within a data factory, look at deploying a second data factory.</p>
</li>
<li>
<p>You can't publish from private branches.</p>
</li>
<li>
<p>You can't currently host projects on Bitbucket.</p>
</li>
</ul>
